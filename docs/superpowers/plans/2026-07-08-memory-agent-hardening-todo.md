# Memory Agent 健壮性待办

> **日期**: 2026-07-08 ｜ **作者**: 小天 ｜ **类型**: 待办清单(非设计文档,不含实现方案)
>
> **关联**: [Graupel Memory 修缮设计](../specs/2026-06-26-graupel-memory-fix-design.md) —— 该设计让 LibreChat 自带的 Memory Agent 对普通用户生效,本文档是在此基础上、通过代码走查发现的**健壮性缺口**,不是新 bug,是现有实现的已知取舍,记录下来供以后决定要不要补。

## 背景

自带 Memory Agent 的三层机制(存储/抽取/注入)本身跑得通,且架构上和 ChatGPT Memories 同构,是业内主流做法。走查 `api/server/controllers/agents/client.js` + `packages/api/src/agents/memory.ts` 代码后,发现三个具体的健壮性短板,均与"要不要上向量检索"这类功能取舍无关,纯粹是可靠性问题。

## 发现 1:抽取失败无重试,静默丢失

- `packages/api/src/agents/memory.ts:720` `finalLLMConfig` 强制 `maxRetries: 0`(覆盖任何配置值),且没有任何应用层重试逻辑包住 `run.processStream`/`processMemory`。
- 抽取这次 LLM 调用如果失败(网络抖动、限流、供应商故障),直接进 catch(`memory.ts:862-867`)记一条 `logger.error`,这轮记忆彻底没了,用户和前端都无感知。
- **不算紧急**:是"这轮没记住"而不是"记错/记串",且下一轮对话大概率能补上(除非用户只说一次就再也不提)。

## 发现 2:并发无锁保护,可能基于过期计数覆盖

- `runMemory` 每轮对话都触发(`client.js:1056-1058`),没有任何跨请求的锁/去抖/排队机制。
- `writeChain` 串行化(`memory.ts:140,244-247`)只保证**单次** `processMemory` 内部多个 `set_memory` 调用不冲突,不保证同一用户**跨请求**并发时不冲突。
- 如果用户连续快速发多条消息,可能出现多次 `runMemory` 并发跑,各自读到独立的 `totalTokens` 快照(`memory.ts:892`)、按各自快照判断要不要写,最后谁写完谁生效——存在基于过期计数覆盖的竞态可能。
- 这是走查代码推断出的风险点,不是有复现用例的确诊 bug。

## 发现 3:失败对用户不可见

- 成功但"无需记住"(`memory.ts:859`)和真失败(`memory.ts:862-867`)都只有 debug/error 级别日志,没有任何用户侧提示区分"这轮没记"是因为不需要记还是抽取挂了。
- 3 秒超时(`awaitMemoryWithTimeout`,`client.js:503-523`)本身不丢数据——`Promise.race` 不取消败者,超时后抽取仍在后台跑完、`set_memory` 仍会落库,只是那次的"记忆已更新" UI 提示不会出现(因为主流程已经不等这个返回值了)。这一点容易被误解成"超时=丢失",特此记录澄清。

## 优先级建议(仅供参考,未决定是否要做)

1. 发现 1(补重试)—— 成本最低,收益明确,失败静默这件事本身值得先解决
2. 发现 3(失败可观测性,至少加个内部监控/告警,不一定要做用户侧 UI)—— 配合发现 1 一起做成本很低
3. 发现 2(并发锁)—— 只有在实际观察到用户高频连续发消息导致记忆覆盖/丢失的具体反馈之后再动,目前是理论风险,没有实测复现

## 明确不做

- 语义/向量检索式记忆取用(参考 Mem0/Zep 那种架构)—— 是完全不同量级的重构,现阶段用户记忆条数少,原样注入够用,不在这份待办范围内。
